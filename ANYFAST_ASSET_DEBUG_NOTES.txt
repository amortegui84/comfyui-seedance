AnyFast asset debug notes
Updated: 2026-04-25 23:16:12 -04:00

Context
- fal.ai works with the current node flow using base64 data URIs for image references.
- AnyFast failed with image references during Seedance generation.

Observed failures
1. Initial AnyFast generation failure:
   - InvalidParameter on content[1].image_url
   - This pointed to AnyFast rejecting direct base64 data URIs in the generation payload.

2. After switching AnyFast runtime images to asset uploads:
   - InvalidParameter on content[2].image_url.url
   - Message: specified asset asset-... is not found
   - This showed the asset upload succeeded, but the generation endpoint could not see the asset yet.

Implemented changes
1. Provider split kept in the same node:
   - fal.ai path unchanged
   - anyfast path uploads first_frame / last_frame / reference_images as temporary assets

2. AnyFast runtime image upload changed to:
   - CreateAsset via JSON
   - URL = data:image/png;base64,...
   - This matches the AnyFast asset-management docs better than multipart for runtime image tensors

3. Added AnyFast asset visibility polling:
   - After CreateAsset returns an asset id, the code now polls ListAssets
   - Goal: wait until the uploaded asset becomes visible before using Asset://... in generation

Important operational notes
- Use one fixed API key during the entire test
- Keep token grouping fixed to Seedance, not Auto Select
- Restart ComfyUI after changing the key/provider setup
- The old custom human-asset node flow was removed from the recommended path
- Official hybrid flow remains the recommended real-human workflow

If testing resumes later
- Re-test AnyFast generation with the same fixed token and grouping
- If it still fails with asset not found, next hypothesis to verify:
  - AnyFast may require a different asset query filter/status check before the asset is truly usable in generation
  - or a different group/type visibility rule for runtime-created image assets
